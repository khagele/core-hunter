# Decision — 3D mode: 2D/3D FAB, extruded hex bars, buildings, terrain (#147)

2026-07-11. Phase 2 of the MapLibre migration (`docs/2026-07-06-maplibre-basemap.md`):
a 2D/3D FAB next to the layer-toggle, following the bars-only scope from the
original spike conclusion but **with buildings and terrain included**, since
both turned out to be key-free (no secrets/config concern) once prototyped.

## What the FAB does

`set3D(v)` in `huntmap.js`:
- Tilts the camera (`pitch` 0 → 60°) via `easeTo`.
- Swaps the flat `hex` fill layer for its `hex-3d` fill-extrusion twin — same
  `hex` GeoJSON source, with a `height` property added per feature.
- Shows/hides the `buildings-3d` layer.

No new gesture: the FAB is the only way to pitch the camera (`pitchWithRotate`
stays `false`, matching the existing rotate-gesture behaviour from #116) — kept
in scope with the original decision ("just toggles pitch"), not a free-tilt
3D explorer.

## Signal bars: height reuses the existing RSSI tiers

`signal.js::extrusionHeight(rssi, offset)` buckets height by the same fixed
dBm bands as `rssiTier`/`tierColorVar`, so a bar's height and colour always
agree on the same tier (a "hot" cell is always both the tallest and the
hottest-coloured). Same `hex` source as the 2D layer; the 3D layer just reads
an extra `height` property the 2D layer ignores.

## Buildings: no new network cost

The OpenFreeMap hosted style (`dark`/`positron`) already ships an
`openmaptiles` vector source with a `building` source-layer, rendered flat at
`minzoom: 12` as part of the ordinary 2D basemap. `buildings-3d` just adds a
`fill-extrusion` layer against that **same, already-fetched** source/layer —
toggling 3D does not trigger any additional building-data request. Colour via
the new `--ch-building` token (dark `#39465a` / light `#cfc7b6`), per the
CSS-tokens hard rule. Only present on the hosted style, guarded by
`map.getSource('openmaptiles')`, so the bare offline fallback style still
renders (no buildings, hex bars only).

## Terrain: prototyped, then dropped

Terrain was prototyped with AWS's public Terrarium raster-dem tiles
(`s3.amazonaws.com/elevation-tiles-prod/terrarium`) but **removed before ship**.
Two reasons, both decisive:

1. **It froze the map.** With terrain enabled the DEM source never settled
   (`map.loaded()` stayed false, tiles perpetually pending), keeping the map in
   a continuous load/render loop. Desktop survived at ~30 fps; on weaker/mobile
   GPUs — the actual hunting device — enabling 3D froze the whole map.
2. **It was the one genuinely new external dependency** (buildings and the
   basemap were already fetched), with no uptime or privacy guarantee from AWS
   beyond "public bucket."

3D mode keeps the cheap, self-contained parts: the camera tilt, the extruded
hex signal bars, and 3D buildings (reused from the already-fetched
`openmaptiles` vector source). No DEM host, no new request, no freeze. This
returns to the bars-first scope of the original spike, plus buildings.

## §7 disclaimer + interaction parity

No new disclaimer surface was needed: the hex/point layers were already
outside the §7 disclaimer's scope (only the Locate estimate readout carries
it, per the existing `SPLASH_DISCLAIMER` usage in `app.js`), and 3D just
re-renders the *same* hex data as extruded bars — it doesn't add a new
location-estimate output. The Locate readout/disclaimer box is DOM-based
(`lc-disclaimer`), unaffected by camera pitch, so it stays visible in 3D
exactly as in 2D. Point-click popups (`points` circle layer) stay flat and
clickable regardless of pitch or what's drawn above them, since MapLibre's
`click`-on-layer binding targets that specific layer's geometry, not
whatever is visually topmost.

## Follow-up fixes (post-review)

Two map-rendering defects turned up while dogfooding 3D:

1. **Blank / corner-only render.** MapLibre latches onto its 400×300 zero-size
   fallback when the map is built before `#map` has laid out (backgrounded tab,
   PWA cold start, or creation after `await loadConfig()` in `DOMContentLoaded`)
   and doesn't reliably recover. `syncSize()` on the render-tick `draw()` path
   reconciles the canvas to its container whenever they disagree (self-heals
   within ~1s).

2. **Terrain froze the map → terrain removed** (see the terrain section above).
   A side benefit: with terrain gone, `easeTo({pitch})` animates normally, so
   `set3D` is back to a plain tilt + layer-visibility swap. (While terrain was
   still in, it had a nastier second bug: `easeTo({pitch})` is a no-op *while
   terrain is enabled*, so tilting and enabling terrain in one tick left the
   camera flat. Dropping terrain moots it.)
