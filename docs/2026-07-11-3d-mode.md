## Signal bars: height reuses the existing RSSI tiers

`signal.js::extrusionHeight(rssi, offset)` buckets height by the same fixed
dBm bands as `rssiTier`/`tierColorVar`, so a bar's height and colour always
agree on the same tier (a "hot" cell is always both the tallest and the
hottest-coloured). Same `hex` source as the 2D layer; the 3D layer just reads
an extra `height` property the 2D layer ignores.

## Buildings: no new network cost

The OpenFreeMap hosted style (`dark`/`positron`) already ships an
`openmaptiles` vector source with a `building` source-layer, rendered flat at
`minzoom: 12` as part of the ordinary 2D basemap. `buildings-3d` just adds a
`fill-extrusion` layer against that **same, already-fetched** source/layer â€”
toggling 3D does not trigger any additional building-data request. Colour via
the new `--ch-building` token (dark `#39465a` / light `#cfc7b6`), per the
CSS-tokens hard rule. Only present on the hosted style, guarded by
`map.getSource('openmaptiles')`, so the bare offline fallback style still
renders (no buildings, hex bars only).

## Terrain: new third-party host (key-free)

Terrain uses AWS's public Terrarium raster-dem tiles
(`s3.amazonaws.com/elevation-tiles-prod/terrarium`) â€” key-free, same posture
note as the OpenFreeMap basemap swap: every 3D-mode pan/zoom sends viewport
bounds to this host, with no uptime or privacy guarantee from AWS beyond
"public bucket." This is the one genuinely **new** external dependency 3D mode
adds (buildings and the basemap itself were already fetched). DEM tiles are
only requested while `mode3D` is true (`setTerrain` is called with `null` on
leaving 3D), so 2D-only sessions never touch this host.
